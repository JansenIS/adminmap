# Дорожная карта интеграции «ходовой» модели (без поломки текущих систем)

## Цели и ограничения

- Сохранять совместимость с текущими `adminmap` + `economy_sim_ui` на переходный период.
- Делать интеграцию по принципу **strangler pattern**: новые модули подключаются рядом, затем постепенно переключают существующие флоу.
- Ввести единый «ход» (`turn/year`) как главный источник времени для экономики, армии, дипломатии и боёв.

## Рекомендуемый порядок внедрения

## Этап 0. Архитектурный baseline (обязательный подготовительный шаг)

Статус: базовая спецификация вынесена в `docs/turn-engine-stage0-baseline.md` и `docs/contracts/turn-engine-stage0.schema.json`.

**Что делаем**
- Фиксируем «каноническую» модель данных: `turn`, `snapshot`, `orders`, `events`, `ledger`, `war`, `diplomacy`.
- Описываем контракты API для чтения/записи, версионирование и политику конкурентных изменений (`If-Match`/version).
- Добавляем таблицу/JSON-спеку зависимостей между доменами (экономика зависит от turn, армия зависит от экономики и дипломатии и т.д.).

**Почему первым**
- Без этого дальнейшие модули будут реализованы с разными представлениями времени и состояния.

**Критерий готовности**
- Документ с доменной схемой и последовательностью вычислений хода принят как «источник правды».

---

## Этап 1. Snapshot/Turn Engine (ваш пункт №1 + база для №6)

Статус: базовая backend-реализация заведена, детали в `docs/turn-engine-stage1-status.md`.

**Что делаем**
- Вводим сущности:
  - `Turn` (номер/год, статус: draft|processing|published|rolled_back).
  - `WorldSnapshot` (срез на начало/конец хода).
  - `MapArtifacts` (обычная карта, гекс-карта, дополнительные слои).
  - `EntityState` и `EconomyState` на конкретный `turn_id`.
- Реализуем операции:
  - `createTurnFromPrevious(turn_n -> turn_n+1)`.
  - `processTurnEconomy(turn_n+1)` синхронно и детерминированно.
  - `publishTurn(turn_n+1)` и API загрузки любого опубликованного хода.
- Удаляем из пользовательского процесса «ручные» reset/simulation настройки экономики: они заменяются параметрами сценария, привязанными к дате/ходу.

**Почему вторым**
- Все остальные пункты (армии, дипломатия, бои, летопись) требуют стабильной оси времени и воспроизводимых срезов.

**Критерий готовности**
- Можно поднять произвольный ход `N` и восстановить карту, сущности и экономику бит-в-бит.

## Этап 2. Казна в сущностях и провинциях (ваш пункт №2)

Статус: базовая реализация treasury + ledger заведена, детали в `docs/turn-engine-stage2-status.md`.

**Что делаем**
- Добавляем двухконтурную финансовую модель:
  - `entity_treasury` (государственная казна).
  - `province_treasury` (локальные фонды/резервы).
- Вводим `TreasuryLedger` (журнал транзакций) с привязкой к `turn_id`, источнику и причине (налоги, дотации, расходы на армию, торговля).
- Добавляем правила трансферов: провинция -> сущность, сущность -> провинция, межсущностные выплаты.

**Почему до армий/дипломатии**
- Призыв, содержание, торговля и контрибуции должны списывать/начислять деньги из одной унифицированной системы.

**Критерий готовности**
- Любое изменение денег объяснимо через журнал и воспроизводимо при перерасчёте хода.

## Этап 3. Дипломатия и торговые договоры (ваш пункт №4)

**Что делаем**
- Реестр договоров: тип, стороны, условия, дата вступления, срок, штрафы.
- Механика эффектов договора как модификаторы экономики (`tax_modifier`, `trade_flow`, `tariffs`, `subsidies`).
- Жизненный цикл: draft -> active -> breached/expired, с событиями для летописи.

**Почему перед войсками**
- Дипломатические статусы (союз, вассалитет, торговые льготы, эмбарго) напрямую влияют на возможность призыва и стоимость снабжения.

**Критерий готовности**
- Экономика хода учитывает активные договоры автоматически.

---

## Этап 4. Система призыва и содержания войск (ваш пункт №3)

**Что делаем**
- Модель сил:
  - собственные войска,
  - вассальные контингенты,
  - сроки службы/ротации,
  - лимиты мобилизации по провинциям.
- Стоимость:
  - единоразовый призыв,
  - содержание за ход,
  - логистика/снабжение,
  - модификаторы от договоров и экономики.
- Orders API для военных приказов с валидацией прав и ресурсов.

**Почему после казны и дипломатии**
- Без финансового леджера и договорных модификаторов система войск будет дублировать правила и ломать баланс.

**Критерий готовности**
- Нельзя издать военный приказ без достаточных ресурсов/прав; расходы прозрачно проходят через казну.

---

## Этап 5. Интеграция боевой системы (ваш пункт №5)

**Что делаем**
- Подключаем combat-модуль как отдельный вычислительный шаг `resolveBattles(turn_n)`.
- Источник входа: подтверждённые приказы + состояние армий на начало шага.
- Выход: потери, изменение контроля провинций, эффекты на экономику и дипломатию следующего шага.

**Почему после армий**
- Бой без нормализованной модели армий и приказов превращается в hardcode и не масштабируется.

**Критерий готовности**
- Результаты боёв детерминированы и воспроизводимы для одинакового snapshot + orders.

---

## Этап 6. Летопись хода (ваш пункт №6)

**Что делаем**
- Вводим `TurnChronicle` (структурированная + человекочитаемая запись).
- Генерация из доменных событий: экономика, дипломатия, мобилизация, бои, административные решения.
- Разделяем:
  - «фактический лог» (неизменяемые события),
  - «нарративный слой» (текстовое описание для публикации).

**Почему после основных доменов**
- Летопись должна агрегировать уже существующие события, а не быть ручным источником данных.

**Критерий готовности**
- По опубликованному ходу формируется автоматическая хроника + экспорт.

---

## Этап 7. Кабинеты игроков + авторизация VK/TG (ваш пункт №7)

**Что делаем**
- Добавляем player-portal:
  - привязка пользователя к одной/нескольким сущностям,
  - подача приказов,
  - публикация ролевых постов,
  - просмотр состояния на доступных ходах.
- Авторизация через бота (VK/TG) как identity-provider + выдача session/token для веб-кабинета.
- RBAC: игрок, модератор, админ.

**Почему ближе к концу**
- UI кабинетов должен опираться на уже стабильные API приказов, экономики и летописи.

**Критерий готовности**
- Игрок может безопасно отправить приказ/пост только в пределах прав своей сущности.

---

## Этап 8. Админка модерации и вердиктов (ваш пункт №8)

**Что делаем**
- Moderation workflow:
  - очередь приказов/постов,
  - статусы `pending/reviewed/approved/rejected/needs_clarification`,
  - вердикты с обязательной причиной,
  - аудит-кто/когда/почему.
- Интеграция с движком хода:
  - в расчёт попадают только `approved` приказы,
  - отклонённые остаются в истории, но не влияют на state.
- Инструменты ручных правок с обязательным event-log.

**Почему последним**
- Модерация — это оркестрация поверх уже созданных доменных модулей.

**Критерий готовности**
- Полный цикл «приказ игрока -> модерация -> расчёт хода -> публикация хроники» работает end-to-end.

## Техническая стратегия «без поломок»

1. **Feature flags на каждый этап**
   - `TURN_ENGINE_V1`, `TREASURY_V1`, `DIPLOMACY_V1`, `ARMY_V1`, `COMBAT_V1`, `PLAYER_PORTAL_V1`, `MODERATION_V1`.
2. **Dual-write / shadow-read на переходе**
   - Сначала пишем в новую модель параллельно старой, сравниваем расхождения.
3. **Idempotent jobs + retry**
   - Все шаги хода должны безопасно перезапускаться.
4. **Versioned snapshots**
   - Жёсткая привязка расчётов к версии правил/формул.
5. **Контрактные тесты API и golden-тесты хода**
   - Проверка воспроизводимости одного и того же хода на одинаковом входе.

## Минимальные релизы (MVP-срезы)

- **MVP-1**: Этапы 0–2 (ход + snapshot + казна).
- **MVP-2**: Этапы 3–4 (дипломатия + армии).
- **MVP-3**: Этапы 5–6 (бои + летопись).
- **MVP-4**: Этапы 7–8 (кабинеты + модерация).

## Главные риски и как снять

- **Риск:** расхождение между старой и новой экономикой.
  - **Митигируем:** shadow-режим на 5–10 ходов и автоматическое сравнение агрегатов.
- **Риск:** недетерминированность боёв.
  - **Митигируем:** фиксированные seed/правила и обязательный replay-тест.
- **Риск:** слишком ранний запуск кабинетов игроков.
  - **Митигируем:** открывать UI только после стабилизации доменных API.
