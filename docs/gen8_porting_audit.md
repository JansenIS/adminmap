# Аудит переноса логики gen8 в `genealogy_simulator`

Сравнение выполнено между:
- эталонным демо-алгоритмом в `genealogy_8gen_example.html` (логика на Union-узлах),
- текущей реализацией в `genealogy_simulator/app.js`.

## Краткий вывод

Перенесена **часть визуальной трассировки** (ортогональные полилинии, семейные rail'ы и разведение горизонталей по трекам), а также **часть поведенческих ограничений** (супруги ближе, группировка детей под родителями).

Не перенесено ядро gen8: **нормализация в Person↔Union-модель как основа layout**, **tidy layout по Union-дереву (measure/place)**, **ширины поддеревьев**, **якорение rootUnion**, **полноценная barycenter-итерация с перераскладкой** и **выход в виде `positions + segments` как отдельный вычислительный слой**.

## Что перенесено / адаптировано

### 1) Ортогональная геометрия линий — перенесено
- В проекте ребра рисуются как `polyline` с вертикально-горизонтальными сегментами (`createPolyline`).
- Для супружеской связи: вертикаль от нижних точек кругов + горизонтальный участок rail.
- Для семьи с детьми: есть «ствол» от marriage midpoint к sibling midpoint, sibling-горизонталь и вертикали детей к этой горизонтали.

Подтверждение:
- эталонная схема в gen8 (`buildEdges`) с брачной линией, стволом, sibling rail и child verticals;
- аналогичная схема в симуляторе при отрисовке `families`.

### 2) Разведение горизонталей по трекам (interval tracks) — перенесено частично, но рабоче
- В gen8 есть `assignIntervalTracks` + применение отдельно к sibling и connector интервалам по поколениям.
- В проекте есть `assignIntervalTracks` и `trackByGeneration`, применяется к:
  - spouse/sibling ребрам одного поколения,
  - семейным sibling rail,
  - межпоколенческому connector rail.

То есть ключевая идея интервального трекинга сохранена.

### 3) Поколения сверху-вниз — перенесено в альтернативной форме
- В gen8 поколение задается через `union.gen` и `person.gen`, `genY(g)`.
- В проекте поколения вычисляются через `computeGenerations` (Union-Find для peer-связей + DAG-проход по `parent_child`) и дальше ставятся на `y = 130 + g*230`.

Это не Union-based BFS как в gen8, но визуально дает слои поколений сверху-вниз.

### 4) Ограничение «супруги рядом» — перенесено эвристикой
- В проекте есть специальные фазы `enforceStrictSpouseAdjacency` и несколько проходов layout, чтобы «прижимать» супругов.
- Есть отдельный `spouseSpacing` меньше общего межузлового `spacing`.

### 5) Группировка детей под семейным блоком — перенесено эвристикой
- В проекте есть `families` (пара родителей или single-parent), сбор детей и отрисовка общей sibling rail.
- В layout есть блоки детей, центровка на owner-unit и последующее разведение блоков.

## Что не перенесено (или перенесено не как в gen8)

### 1) Нормализация входа в `UnionNode` как основной граф layout — **не перенесено**
- В gen8 layout базируется на явных `unions` с полями `partners`, `children`, `gen`, `anchorX`, `subtreeW`.
- В проекте доменная модель API — только отношения `parent_child|spouses|siblings` по `source_id/target_id`, без `unionId`.
- Семейные «юниты» в проекте собираются временно на рендер-этапе (`families`), но это не структурный граф layout.

### 2) `unionId`/множественные браки как отдельные союзы — **не перенесено**
- В требуемой модели ключевое поле `unionId` обязательно для различения браков.
- В текущем API и данных `unionId` отсутствует, поэтому невозможно нативно различить несколько союзов одной и той же пары/персоны на уровне данных.

### 3) Tidy layout по Union-дереву (measure/place) — **не перенесено**
- В gen8 есть двухфазная схема `measureUnion` (subtree width) + `placeUnion` (anchorX и рекурсивная раскладка).
- В проекте используется итеративный force/constraint-подобный heuristic layout (`relaxRow`, блоки родителей/супругов/сиблингов), без расчетов `subtreeW` по Union-дереву.

### 4) Центрирование union относительно диапазона детей как базовый инвариант — **не полностью**
- В gen8 это базовая формула (`anchorX` и child block around anchor).
- В проекте похожий эффект достигается эвристиками центров блоков/owner-unit, но нет явного `union.anchorX` как первичного узла layout.

### 5) Полноценная barycenter-оптимизация (итерации сверху-вниз с перераскладкой) — **почти нет**
- В gen8 есть явная `reorderChildrenOnce` по barycenter дочерних union/person, затем повторный measure/place.
- В проекте есть много итераций релаксации, но именно barycenter-переупорядочивание детей как отдельный шаг не выделено.

### 6) Обработка циклов через reference-node/dedup — **не реализована как в требованиях**
- В проекте для циклов в `parent_child` есть «мягкая защита» (ограниченный релакс/игнор части противоречий), но нет явного создания reference-node.

### 7) Выход алгоритма как чистые данные (`positions` + `segments`) — **частично**
- В gen8 есть отдельный шаг `buildEdges`, выдающий набор сегментов.
- В проекте сегменты строятся и сразу рисуются в DOM, нет явного экспортируемого слоя `polylines/segments` как API результата layout.

## Итог по вашему списку требований

- **Перенесено:**
  - ортогональная отрисовка семейных связей,
  - sibling rail + child verticals,
  - интервальные треки для предотвращения наложений горизонталей,
  - слои поколений top-down,
  - эвристика «супруги рядом».

- **Не перенесено (ключевое):**
  - Union-first модель данных (`unionId`-центричная),
  - tidy/RT-подобный layout на дереве Union,
  - расчет ширин поддеревьев и жесткое центрирование anchor по детям,
  - barycenter-реордер с перераскладкой как отдельный этап,
  - reference-node стратегия для циклов/дубликатов,
  - структурированный output `positions + segments` как независимый результат.

## Рекомендация по следующему шагу

Если цель — именно поведение «как в gen8» (включая пирамидальность на сложных кейсах и множественные браки), минимально нужно:
1. расширить модель отношений полем `unionId`,
2. построить слой нормализации `Person↔Union`,
3. перенести `measure/place` (union subtree width + anchor),
4. сохранить текущий track-router для горизонталей как post-process.


## Прогресс переноса (после текущих изменений)

Сделано дополнительно:
- Legacy post-pass для X-раскладки полностью отключен: итоговую X-геометрию задает family-slot/tidy этап.
- Добавлена автоматическая миграция старых relationships в `genealogy_load()` (дозаполнение `union_id`/`parents_union_id`).
- В API relationships добавлены поля `union_id` и `parents_union_id`; сравнение дублей теперь учитывает их для `spouses`/`parent_child`.
- В layout добавлен первичный сбор cycle-reference маркеров для конфликтных parent_child ребер.
- Добавлен жесткий anchor-инвариант: после слот-раскладки child-range центрируется относительно `anchorX` семьи.
- Включена многопроходная barycenter-итерация (top-down + bottom-up, 2 pass) с промежуточной перераскладкой слотов.
- В `genealogy_simulator/app.js` добавлена промежуточная family-slot раскладка: семейные блоки, `subW` снизу вверх, слоты `requiredW(child)` и push-проход непересекаемости.
- Добавлен первый barycenter-проход: дети внутри семьи переупорядочиваются по центрам их дочерних семей, после чего выполняется повторный пересчёт `subW` и повторная slot-раскладка.
- В рендере семейные связи (`families`) теперь по возможности берутся из результатов layout (`state.layoutFamilies`), чтобы sibling rail строились **по конкретной семье**, а не пере-собирались только из сырых ребер.

Что осталось сделать дальше (ключевое):
1. Проверить и откалибровать визуальный результат после полного отключения legacy post-pass на реальных больших данных (возможны edge-cases spacing).
2. Миграция старых данных в backend добавлена; осталось обновить UI-формы админки для явного задания/редактирования `union_id` и `parents_union_id`.
3. Взаимодействие `anchorX`-инварианта с push-проходом усилено повторной центровкой; осталось подобрать константы смещений для минимизации дрожания при многократных pass.
4. Тонко настроить веса/критерии barycenter, чтобы стабилизировать порядок на сложных графах с несколькими браками.
5. Reference-маркеры уже визуализируются (пунктир + label); осталось добавить интерактивность/легенду и при необходимости отдельный тип pseudo-node.
